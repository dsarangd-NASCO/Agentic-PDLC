# ADR-002: Use Single PostgreSQL Instance (No Sharding, No Aurora)

- **Date:** 2026-05-21
- **Status:** Proposed
- **Ticket:** PDLC Stage 2 Design
- **Supersedes:** None
- **Superseded by:** (Future scaling may warrant ADR for sharding)

---

## Context

### Platform Standard (Default Decision)

The NASCO platform engineering standards (copilot-instructions.md §16) default to:
> "Database: Single PostgreSQL instance — polyglot persistence requires ADR"

This aligns with the monolith-first principle: deploy-hub starts as a modular monolith with a single persistent data store.

### Deploy-Hub's Database Requirements

Deploy-hub must store:

1. **Deployment State** (immutable append-only log)
   - deployment_id, service_id, artifact_url, status, created_at, updated_at, etc.
   - ~1–2 KB per record; estimated 100+ deployments/day = 100–200 KB/day
   - 3-month retention (90 days per SPEC.md compliance requirement)
   - ~5.4 MB total (100 × 1.5 KB/day × 90 days)

2. **Deployment Stages** (execution logs)
   - deployment_id, stage_name, logs (JSONL, potentially large)
   - Logs ~100 KB per deployment stage (5 stages × 20 KB/stage)
   - 3-month retention: 100 deployments/day × 500 KB = 50 MB/day = 4.5 GB/quarter

3. **Service Config** (reference data)
   - service_id, artifact_registry_url, health_check_url, rollback_strategy
   - ~500 bytes per service × 50–100 services = 25–50 KB total
   - Static data; grows slowly

4. **Audit Logs** (immutable compliance record)
   - Per SPEC.md: "All deployment events immutable and queryable for ≥ 90 days"
   - ~1 KB per audit record × 500+ events per deployment = 500+ KB/deployment
   - 100 deployments/day × 500 KB = 50 MB/day = 4.5 GB/quarter

**Total Storage:** ~5 GB per quarter (MVP scale)

### Query Patterns

| Query | Frequency | SLA | Access Pattern |
|---|---|---|---|
| Get deployment status | 100 req/min (per service, 10 services active) | <100 ms | `SELECT * FROM deployments WHERE id = ?` — cached or indexed |
| List deployment stages | 10 req/min | <200 ms | `SELECT * FROM deployment_stages WHERE deployment_id = ?` |
| Audit trail query | 5 req/day | <1 sec | `SELECT * FROM audit_logs WHERE deployment_id = ? AND event_at BETWEEN ? AND ?` |
| Deployment history | 20 req/day | <2 sec | `SELECT * FROM deployments WHERE service_id = ? ORDER BY created_at DESC LIMIT 30` |

**Observation:** All queries are lookup/scan operations; no complex joins. Deployments are independent entities; no distributed transactions.

## Decision

**Deploy-hub will use a single PostgreSQL instance (AWS RDS PostgreSQL 15+) with no sharding and no Aurora.**

### Implementation Details

1. **Database Topology**
   - **Instance Type:** db.t3.small (2 vCPU, 2 GB RAM) for MVP
   - **Storage:** 100 GB gp3 (3000 IOPS baseline, auto-scaling enabled)
   - **Multi-AZ Failover:** Enabled (synchronous replica in standby AZ)
   - **Automated Backups:** Daily snapshots, 30-day retention (can restore to any point in time)
   - **Encryption:** At-rest (RDS encryption), in-transit (SSL/TLS)
   - **Network:** Private RDS subnet; access via ECS security group only

2. **Schema Design** (see DESIGN.md Data Model)
   - Immutable append-only log pattern (no UPDATE on core deployment records)
   - Indexes on frequently-queried columns: `deployments(id)`, `audit_logs(deployment_id, event_at)`
   - Partitioning by month (future optimization if 4.5 GB/quarter grows beyond RDS capacity)

3. **Backup & Recovery**
   - RDS automated snapshots: one per day, 30 days retention
   - Deploy-hub can restore to point-in-time within 30-day window
   - Compliance requirement (SPEC.md): 90-day audit log retention; separate S3 archive for older logs

4. **Scalability Path**
   - **Phase 1 (MVP, Weeks 1–8):** db.t3.small, 100 GB storage
   - **Phase 2 (Growth, Month 3–6):** Scale to db.t3.medium (2 vCPU, 4 GB RAM, 200 GB storage) if CPU/memory utilization exceeds 70%
   - **Phase 3 (Advanced):** If query latency degrades, add read-only replica (db.t3.small) for audit queries; deploy-hub connects to read replica for long-running compliance queries

5. **Connection Management**
   - NestJS TypeORM connection pool: 10 connections (default)
   - PgBouncer (optional MVP+1): 20 connections, transaction mode
   - Max concurrent connections: 20 (comfortable headroom for 2–5 ECS tasks)

## Why Not Aurora?

### Aurora Advantages (Not Needed for MVP)

- Multi-master replication
- Auto-scaling storage
- Advanced failure recovery (sub-second failover)
- Native data lake integration

### Why Single PostgreSQL Is Better for MVP

1. **Cost:** RDS PostgreSQL db.t3.small ~$50/month; Aurora minimum ~$100/month. 2x cost reduction.

2. **Operational Simplicity:** Single instance is easier to troubleshoot. Aurora's complexity is only beneficial if you need multi-master writes or rapid scaling (deploy-hub does not).

3. **No Complex Migrations:** Aurora requires different connection strings, connection pooling strategy, and failover handling. Single PostgreSQL has well-known operational patterns.

4. **Sufficient Availability:** Multi-AZ failover (built-in RDS feature) meets 99.5% SLA requirement (see DESIGN.md deployment model).

5. **Append-Only Pattern Fits Well:** Deployments are written once, never updated (except status). Aurora's read-scaling advantage is diminished; single write-master is sufficient.

### Scaling Path: When to Consider Alternatives

| Scenario | Recommendation | Timing |
|---|---|---|
| **>100 GB data; >1000 QPS** | Sharding by service_id → separate DB per logical partition | Month 9+ (requires ADR) |
| **Read-heavy audit queries block deployments** | Read replica (Aurora or manual logical replication) | Month 6+ (simple RDS read replica sufficient) |
| **Multi-region deployment coordination** | Aurora global database or DynamoDB for cross-region consensus | Month 12+ (requires ADR) |
| **Real-time analytics (OLAP)** | Separate OLAP DB (Redshift, BigQuery); PostgreSQL remains OLTP | Month 9+ (separate ADR) |

## Why Not Managed Alternatives (DynamoDB, DocumentDB)?

### NoSQL Considered

| Option | Why Not Selected |
|---|---|
| **DynamoDB** | Deployment state requires strong consistency and ACID transactions; DynamoDB eventual consistency requires complex compensating transaction logic. Audit logs MUST be ordered; DynamoDB sort keys are insufficient. |
| **DocumentDB** | MongoDB-compatible but no better at deployment state management than PostgreSQL. Additional operational overhead (AWS-proprietary replica behavior). |
| **ElastiCache (Redis)** | In-memory only; deployment state would be lost on restart. Redis persistence (RDB/AOF) is not ACID-compliant. Acceptable for ephemeral task queue (future), not core state. |

**Conclusion:** Relational database with ACID transactions is mandatory for deployment state correctness.

## Consequences

### Benefits

1. ✅ **Cost Efficiency:** RDS PostgreSQL is 50% of Aurora cost for MVP scale.
2. ✅ **Operational Simplicity:** Single instance; well-documented RDS operations; standard PostgreSQL tooling.
3. ✅ **ACID Transactions:** Strong consistency guarantees for deployment state updates (no lost deployments, no partial state).
4. ✅ **Append-Only Pattern:** PostgreSQL excels at immutable log storage (natural fit for audit trails).
5. ✅ **Standards Compliance:** Platform standard; no deviation required for core DB.
6. ✅ **Familiar Tooling:** Team knows PostgreSQL; no new database paradigm to learn.

### Costs & Tradeoffs

1. ⚠️ **Single Point of Failure (at Instance Level):** If RDS instance fails, deploy-hub is unavailable. However, mitigation:
   - Multi-AZ failover handles most failures automatically (synchronous replica in standby AZ)
   - RDS automatic backup + point-in-time recovery ensures data durability
   - Deploy-hub outage does NOT block manual deployments (SPEC.md: "Deploy-hub failure does not block manual fallback")

2. ⚠️ **Vertical Scaling Only (MVP):** Cannot scale write throughput horizontally without sharding (future ADR required if write volume exceeds PostgreSQL's single-writer capacity).
   - **Realistic Threshold:** PostgreSQL single instance can handle 1000+ QPS (50 concurrent deployments × 20 QPS = 1000 QPS well within limits)
   - **Mitigation:** Monitor CPU/memory; scale to db.t3.medium if needed. Horizontal scaling (sharding) only needed if we exceed 10,000+ QPS.

3. ⚠️ **Storage Growth:** 4.5 GB/quarter; 36 GB/year. RDS gp3 can handle up to 64 TB, but management overhead increases.
   - **Mitigation:** Implement 90-day compliance retention policy; archive older logs to S3. Re-evaluate after 6 months if retention needs change.

4. ⚠️ **No Multi-Region Native Support:** For future multi-region deployments, deploy-hub will need a separate coordination mechanism (future ADR).
   - **Mitigation:** MVP is single-region. When multi-region required, evaluate DynamoDB global tables or multi-master replication separately.

### Rollback Path If This Decision Fails

If single PostgreSQL proves insufficient:

1. **Option A (Read Scaling):** Add RDS read replica; route audit/compliance queries to read replica. Keeps single-writer consistency.

2. **Option B (Sharding):** Partition by service_id into multiple PostgreSQL instances. Requires ADR and significant refactoring of application layer.

3. **Option C (NoSQL):** Migrate to DynamoDB for deployments + separate SQL database for audit logs. Adds operational complexity but enables horizontal scaling.

**Probability of Failure:** Low. PostgreSQL single instance can easily handle the projected MVP scale (5 GB data, <1000 QPS).
