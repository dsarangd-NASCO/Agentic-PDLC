# ADR-001: Adopt AWS CodePipeline + CodeBuild + CodeDeploy over GitHub Actions

- **Date:** 2026-05-21
- **Status:** Proposed
- **Ticket:** PDLC Stage 2 Design
- **Supersedes:** None
- **Superseded by:** (Future ADRs may replace this if multi-cloud strategy emerges)

## Context

### Platform Standard (Default Decision)

The NASCO platform engineering standards (copilot-instructions.md §16) default to **GitHub Actions** for CI/CD:
> "CI/CD: GitHub Actions"

This default applies to all services unless explicitly deviated via ADR.

### Deploy-Hub's Deviation

Deploy-hub uses **AWS CodePipeline + CodeBuild + CodeDeploy** instead of GitHub Actions.

### Why This Deviation Is Necessary

#### 1. **Native AWS Service Integration (Operational Efficiency)**

Deploy-hub's core responsibility is orchestrating deployments via **CodeBuild** and **CodeDeploy**, not just delegating to GitHub Actions. Deep integration with AWS deployment services is essential because:

- **CodeDeploy** manages traffic cutover, instance draining, and rollback — core deployment orchestration
- **CodeBuild** executes environment-specific validation scripts (schema migrations, health checks, config validation)
- **CodePipeline** provides managed workflow execution with built-in failure handling and retry logic

GitHub Actions cannot natively invoke these AWS services without custom glue code. Using CodePipeline to orchestrate CodeBuild → CodeDeploy creates a **single control plane** for all deployment decisions.

**Example:** If a health check fails post-deployment, deploy-hub must immediately halt CodeDeploy and trigger rollback. CodePipeline's native integration with CodeDeploy allows this synchronously; GitHub Actions would require an additional webhook bridge.

#### 2. **State Machine Reliability (Deployment Safety)**

Deployments are long-running processes (45 minutes) with many state transitions (queued → validating → deploying → verifying → complete/rolled_back).

CodePipeline's **managed state machine** provides:
- Automatic failure recovery (built-in retry with exponential backoff)
- Idempotent stage re-execution (safe to retry failed stages)
- Atomic transitions (no partial state)
- CloudTrail audit trail (immutable event log per deployment)

GitHub Actions is designed for **short-lived CI jobs** (10–30 minutes), not **long-running deployment orchestration**. Re-implementing a reliable state machine in GitHub Actions would duplicate CodePipeline's logic and introduce bugs.

#### 3. **Cost Model (Deployment Transparency)**

- **CodePipeline:** ~$1/execution per pipeline + build/deploy action costs
- **GitHub Actions:** $0.008/minute for Ubuntu runner + storage costs

Deploy-hub expects 50+ concurrent deployments daily (SPEC.md §scalability). Cost difference:

```
Daily deployments: 100
Duration per deployment: 45 minutes (25 in CodeBuild, 15 in CodeDeploy, 5 overhead)

GitHub Actions cost:
  = 100 deployments × 45 min × $0.008/min = $36/day = $1,080/month
  (Assumes runners are dedicated; sharing runners with CI adds contention)

CodePipeline + CodeBuild + CodeDeploy cost:
  = 100 executions × $1 (CodePipeline) = $100 + build costs (CodeBuild)
  = ~$200–300/month (with build minutes included)

Result: CodePipeline is 3–4x cheaper for high-volume deployment orchestration.
```

Additionally, **CodePipeline costs are transparent:** each deployment execution is a discrete billable unit, making cost attribution per service straightforward. GitHub Actions billing is opaque (total runner hours across all jobs).

#### 4. **Separation of Concerns (Team Ownership)**

- **CI (code build):** Owned by platform-eng team; runs in GitHub Actions (remains default for app builds)
- **CD (deployment orchestration):** Owned by infra-core / deploy-hub team; runs in CodePipeline

GitHub Actions is optimized for "build code → run tests → push artifact". Deployments are a different concern (artifact-driven state machine, not code-triggered CI).

This separation allows:
- Platform engineers to own CI processes (GitHub Actions, code review gates)
- Infra team to own deployment processes (CodePipeline, CodeBuild validation, CodeDeploy rollout)
- Clear API boundary between CI and CD (artifact → deployment submission)

#### 5. **AWS-Native Observability & Compliance**

Deploy-hub must provide immutable audit trails (SPEC.md §audit/compliance).

CodePipeline provides:
- **CloudTrail:** Immutable, centralized log of every stage execution, with IAM principal, timestamp, and result
- **CloudWatch:** Detailed execution metrics (stage duration, error rates)
- **AWS Config:** Compliance tracking of deployment configurations

GitHub Actions logs are tied to GitHub and require export/archival for long-term compliance. CodePipeline integrates natively with AWS compliance infrastructure.

## Decision

**Deploy-hub will use AWS CodePipeline + CodeBuild + CodeDeploy for all deployment orchestration.**

### Implementation Details

1. **Deployment Submission Flow**
   - Engineer submits via deploy-hub REST API
   - Deploy-hub API stores deployment metadata in PostgreSQL
   - Deploy-hub orchestrator triggers CodePipeline execution
   - CodePipeline stages:
     - **Stage 1 (Prepare):** CodeBuild job fetches artifact, runs pre-deploy validation
     - **Stage 2 (Deploy):** CodeDeploy action rolls out artifact to ECS
     - **Stage 3 (Verify):** CodeBuild job runs post-deploy health checks
     - **Stage 4 (Finalize):** CodeBuild job logs completion, signals back to deploy-hub

2. **State Synchronization**
   - Deploy-hub polls CodePipeline API every 30 seconds for execution status
   - State transitions (queued → validating → deploying → complete) mapped to CodePipeline stage results
   - If CodePipeline execution fails, deploy-hub triggers rollback via CodeDeploy

3. **Monitoring & Alerting**
   - CloudWatch Events (EventBridge) forwards CodePipeline execution events to deploy-hub
   - Deploy-hub exports deployment metrics to Prometheus (duration, success rate)
   - CloudTrail logs all deployments for compliance (90-day retention)

4. **CI Remains in GitHub Actions**
   - Application builds (compile, unit tests, container image build) → GitHub Actions
   - Artifact pushed to ECR with git SHA tag
   - Deploy-hub pulls artifact by tag and submits to CodePipeline
   - GitHub Actions and CodePipeline have clean API boundary (artifact URL)

## Consequences

### Benefits

1. ✅ **Operational Efficiency:** Native CodePipeline integration eliminates glue code; deploy-hub can focus on business logic (health checks, rollback, multi-region coordination)
2. ✅ **Reliability:** CodePipeline's managed state machine reduces bugs in deployment orchestration; idempotent stages support safe retries
3. ✅ **Cost Transparency:** CodePipeline execution costs are per-deployment (not per-minute runner overhead); easier cost attribution
4. ✅ **Compliance:** CloudTrail + CloudWatch provide audit trails natively; SPEC.md compliance requirements met without custom logging
5. ✅ **Scalability:** CodePipeline supports parallel region deployments; fan-out/fan-in patterns built-in

### Costs & Tradeoffs

1. ⚠️ **AWS Lock-In:** Deploy-hub is now tightly coupled to AWS CodePipeline, CodeBuild, CodeDeploy. Migrating to multi-cloud (GCP, Azure) would require re-architecting orchestration (future ADR required).
   - **Mitigation:** SPEC.md current assumptions are AWS-only (MVP stage). Multi-cloud is a future phase; if required, a separate orchestration layer can abstract cloud-specific APIs.

2. ⚠️ **Team Learning Curve:** Infra team must learn CodePipeline + CodeBuild + CodeDeploy (new tools relative to GitHub Actions).
   - **Mitigation:** AWS services are well-documented; team already uses CodeDeploy for app deployments; training is 1–2 sprints.

3. ⚠️ **Operational Overhead:** CodePipeline requires IAM role management, VPC networking configuration, and CloudWatch alarm setup.
   - **Mitigation:** Terraform IaC templates provided (infra-deploy-hub repository) automate setup; deployment is infrastructure-as-code.

4. ⚠️ **Debugging Complexity:** CodePipeline execution logs span multiple services (CodeBuild logs in CloudWatch, CodeDeploy logs on EC2 instances). Debugging requires multi-service log aggregation.
   - **Mitigation:** Deploy-hub API exposes unified `/deployments/{id}/logs` endpoint; aggregates CloudBuild, CodeDeploy, and application logs into single stream.

### Rollback Path If This Decision Fails

If CodePipeline proves unreliable or insufficient:

1. **Option A (Partial):** Replace CodePipeline with SQS-driven task queue (Temporal, Apache Airflow) while keeping CodeBuild + CodeDeploy for execution. Reduces AWS lock-in for orchestration while preserving native AWS action execution.

2. **Option B (Full):** Replace CodePipeline + CodeBuild + CodeDeploy with GitHub Actions by implementing a custom state machine in deploy-hub (NestJS async event handlers + PostgreSQL state). Deploy-hub would orchestrate GitHub Actions job submissions and poll for completion. Cost would increase 3–4x.

3. **Option C (Multi-Cloud):** Introduce an abstraction layer (Spinnaker, ArgoCD, Flux CD) that abstracts cloud-specific deployment services. Increases complexity but enables multi-cloud support. Requires ADR.

**Probability of Failure:** Low. CodePipeline is AWS's managed orchestration service; it's battle-tested for complex multi-stage deployments. Sufficient if requirements remain AWS-only.
